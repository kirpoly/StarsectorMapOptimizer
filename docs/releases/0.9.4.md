# Отчёт о выпуске 0.9.4

Дата: 2026-07-18.

`0.9.4` объединяет параллельно разработанный commodity-temporal active set с одновременно
интегрированными direct expiry-aware `MutableStatWithTempMods` hybrid и persistent Economy/Market snapshots.
Поставка по-прежнему использует один startup-javaagent и один transactional structural transformer:
второй agent и whole-class replacement не требуются.

## Основная цель

На больших сохранениях `Market.advance()` может занимать значительную долю campaign frame. Vanilla
для каждого commodity на каждом market tick безусловно выполняет:

```text
4 × MutableStatWithTempMods.advance(days)
1 × CommodityOnMarket.reapplyEventMod()
```

Даже когда все четыре temporary-mod maps пусты и relevant stats не менялись. Новый
`patch.commodityTemporalFastPath` хранит private transient ordered active set и полностью исключает
стабильные commodities из этого цикла.

## Объединённая схема

### Ordered Market active set

Точный vanilla `Market` получает private transient state с entries в порядке live commodity list и
отдельным compact active list. На первом tick entry проверяется полностью. После этого stable entry
не выполняет ни четыре stat advances, ни event-mod reapply.

Структурное изменение commodity list определяется немедленно по identity/size/endpoints и полностью
проверяется bounded audit. Первый audit deadline детерминированно разнесён между markets, чтобы
тысячи рынков не выполняли полный контроль на одном frame.

### Dirty tracking через MutableStat

Точный vanilla `MutableStat` получает два private transient synthetic поля owner/role. Четырнадцать
штатных mutation methods имеют predicted-null guard; unbound stats платят только field-read/branch,
а связанные commodity stats немедленно ставят dirty bit. Покрыты `setBaseValue`, `applyMods`,
`modify*` и `unmodify*`, включая temporary-mod additions/removals через базовые paths.

Private source `eMod`, которым владеет `CommodityOnMarket.reapplyEventMod()`, игнорируется как
self-write. Иначе entry оставался бы активным бесконечно.

### Direct expiry scheduler остаётся авторитетным

Активный commodity использует direct scalar scheduler, интегрированный в этом же выпуске:

- repeated `float countdown -= days` для ближайшего expiry;
- one-pass materialize/removal/rebuild на deadline;
- tied-minimum count;
- sync перед mutation/read/save;
- retained vanilla path после public `getMods()`, для subclasses и anomalies.

Включение `patch.commodityTemporalFastPath` автоматически устанавливает этот scheduler даже если
отдельный `patch.tempModExpiryScheduler` случайно выключен: active set не запускается поверх
неподготовленного класса.

### Bounded audit и прямые изменения модов

Обычные API mutations видны на следующем `Market.advance()`. Мод, который сохраняет live map из
`getMods()` и позднее меняет её напрямую, переводит конкретный stat на vanilla scheduler, но market
active set может не увидеть новое содержимое до `commodity.temporalAuditFrames`. Audit будит entries,
материализует/rebuild'ит scheduler state через private bridge и снова определяет active membership.

Shared stat objects, повторное использование одного stat в нескольких ролях, subclasses,
неожиданные maps и helper failures используют полный vanilla loop.

## Дополнительные hot-path меры

- High-volume counters `Markets/Entries/Active/InactiveSkips` обновляются выборочно раз в 64 market
  calls и масштабируются обратно. Rare dirty/reapply/audit/rebuild/fallback counters остаются точными.
- Во время active processing четыре temp maps читаются один раз до advances; повторно проверяются
  только stats, которые действительно были активны и могли опустеть на expiry.
- Audit phases разных markets staggered по identity и campaign generation.

## Намеренные изменения поведения

Это aggressive patch. Приняты следующие отклонения ради производительности:

- direct live-map/list mutation в обход API может быть видна только на bounded audit;
- commodity, ставший dirty из callback другого commodity, может обработаться на следующем market
  tick вместо более поздней позиции того же исходного list pass;
- промежуточное private `timeRemaining` остаётся deferred до sync point по принятому hybrid-контракту.

Не меняются condition, industry, submarket, person, memory и construction callback cadence.
Порядок commodities, реально обработанных в одном tick, совпадает с live list. Safe profile оставляет
active set выключенным.

## Save и API

Все новые поля private/transient/synthetic. XStream smoke проверяет отсутствие:

```text
smo$commodityTemporalState
spp$commodityTemporalOwner
spp$commodityTemporalRole
spp$tempModHybrid*
```

в XML/save. Public/protected API snapshot не меняется. После load state строится лениво.

## Проверка

Release gate на приложенной `Starsector 0.98a-RC8` включает:

- structural positive/negative/idempotency/ownership tests;
- ASM Analyzer и `BasicVerifier` всех transformed concrete methods;
- actual-javaagent smoke с `Market`, `CommodityOnMarket`, `MutableStat` и
  `MutableStatWithTempMods`;
- stable skip, API dirty wake-up, exact expiry, permanent mutation и retained-map bounded audit;
- dependency smoke: active set включён, отдельный temp-mod switch выключен;
- directed float fixtures и randomized expiry regression;
- actual XStream round-trip с materialized durations и отсутствием transient bindings;
- persistent economy, lifecycle/GC, loading/save, hyperspace и startup regressions.

Графический клиент Starsector в build environment не запускался. Численный эффект следует проверять
на реальном сохранении по JFR `Market.advance`, sampled `commodityTemporal*` counters и общей доле
campaign frame.

Полный список переключателей находится в [`PATCHES.md`](../PATCHES.md), правила совместимости — в
[`COMPATIBILITY.md`](../COMPATIBILITY.md), а игровая матрица — в
[`VALIDATION.md`](../VALIDATION.md).
